# 自然語言即計算：JIT / OS / 分散式三軸，同源於可逆性邊界

> 2026-06-27 討論。draft，design/theory 型。
> 怎麼冒出來的：從「SDD 教學」聊開，使用者推出一串系統類比（SDD=編譯、vibe=直譯、knowie=熱點偵測），要求拿掉教學情境、純技術整理。核心收穫不是任一個類比，是**三個獨立類比畫出同一條邊界**。

## 一句話
自然語言正成熟為一種**計算基底**，它的成熟在重演程式系統的歷史；把它拆成一台機器的三條軸——時間（怎麼編譯）、資源（怎麼排程）、空間（怎麼一致）——三軸的分界線疊起來是**同一條：可逆性邊界**。

## 軸一·執行模型（時間）：NL 的 JIT
- 直譯 ＝ vibe coding（便宜即時、無持久 artifact、適合冷路徑）；AOT 編譯 ＝ SDD（昂貴、產耐久結構化 artifact、值得用在會跑很多次/要緊的）。JIT 需前兩者先成熟 → NL 同時有了 SDD + vibe ＝ 具備 JIT 前提。
- **熱點偵測 / PGO ＝ capture（偵測復現）+ consolidate（付代價結晶）**。開發內迴圈就是 JIT runtime：討論＝直譯執行 → capture＝profiling 事件 → consolidate+roadmap＝編譯佇列 → 每個 roadmap 項的 SDD＝真編譯 → 結果回流 experience＝reprofile。`experience.md` ＝ profile 資料庫。
- **trace-JIT 對應**：「每個 loop 觸發 micro-SDD」字面就是 trace-JIT 專編熱*迴圈*。**deopt 對應**：draft 預設衰減、superseded 不刪只標記。連反向操作都映得到 → 不是硬湊（見 [記憶系統](../concepts/記憶系統.md)、[draft/README](README.md) 衰減模型）。

**兩個墓碑（最濃的 why 在「不像」處）：**
- **是 PGO 不是全自動 JIT**：冷熱判斷不是機器 profiler 全自動，consolidate 要人確認 → profiler 是人。這是「人保定案」的必然，別講成全自動。
- **語義等價轉移不過來**：JIT 編「同一支程式只是更快」；SDD spec 不是 vibe code 的加速版，是更 rich 的不同物件。轉移過來的是*機制*（偵測熱點→結晶→攤提），不是「再編譯」。它是 JIT-flavored 的「**意圖晉升為 artifact**」。

## 軸二·資源模型（OS）：NL 計算的作業系統
因果不是「語言之後剛好有 OS」，是**語言製造出它解不了的問題——資源競爭——OS 是那筆債**。很多人+很多 AI 用自然語言在同一基底並行計算，同一筆債回來（誰優先、怎麼排、記憶怎麼不打架、實驗怎麼隔離、狀態怎麼持久共享）。knowie 在補這層：

| OS primitive | knowie |
|---|---|
| Process / 隔離 | `draft/`、branch、worktree（短命可丟） |
| Scheduler | `next` + roadmap（ready queue；人定案＝主控開關） |
| 虛擬記憶體 | 三檔（底下再亂，讀者拿到連貫 view；投影＝位址轉譯，見根公理一） |
| 檔案系統 / 命名空間 | `knowledge/` in git + `[]()`（選 path 不選 resolver＝延伸原則 8） |
| Syscall 介面 | 六個 skill（human+AI ↔ 知識基底的穩定 API） |
| POSIX vs kernel | [協議非平台](../concepts/協議非平台.md)（定義介面不定義實作） |
| VFS / FUSE | 北極星掛載：knowie＝VFS（管 why），LLM Wiki/向量庫＝掛載的 FS（管 what 大規模檢索） |
| IPC | handshake 注入 AGENTS.md（讓彼此看不見的 AI process 共享 why） |
| Kernel / user mode | **不可逆要人 / 可逆自動**（延伸原則 5） |

**預測力是這面鏡子的真價值**（呼應 experience「外部框架是照出 stated-why⊊real-why 的鏡子」並把事後偵測升級成演繹預測）：事後解釋了為何選 path、為何協議非平台；事先指出 FUSE＝VFS，並指出**還沒造、但 OS 必逼你造的東西＝並行控制（lock / transaction / cache coherence）**＝軸三。

## 軸三·一致性模型（空間）：多節點共識
多 agent / 多 repo 一起寫 → CP（強一致）or AP（最終一致）？**knowie 的根本原則其實已經選了 AP，且已把 AP 變安全。**
1. **底層是 git，git 本就 AP、多 master、最終一致**（分支發散＝容忍分區，merge＝最終收斂，無全域鎖）。寄生其上即繼承其 CAP 立場。
2. **延伸原則 6 + principles 裡那行 CALM ＝ 讓 AP 安全的機制**。CALM：單調程式無需協調。「append、superseded 不刪、history 因果偏序」刻意把 why-log 做成**單調** → 可無協調最終一致。兩節點 history merge ＝ 取**聯集 + 因果排序**；衝突不是覆蓋，是「兩條平行轉移都記下」。已是 event-sourced、近 CRDT 的 log（見 [意義可追溯改寫](../concepts/意義可追溯改寫.md)）。

**但 AP 海裡有一座 CP 島：不可逆的人類決定**（定案、改根公理、認母概念）需在某時點達成協議，不能各自最終一致。正解＝**按類別混合**：
- **基底層（AP，零協調）**：capture / draft / episode / concept / history event → 聯集 merge，CALM 保證安全。
- **共識層（CP，要協議）**：三檔 committed 狀態 / 根公理 → 一輪 coordination（最樸素版：**人閘控的 merge-to-main 就是那輪共識**）。

技術形狀＝**git federation**：branch＝本地發散，PR/merge＝一輪共識，main＝被批准的 view。

## 收束：三軸同一條邊界 ⭐（這是本檔的核心，母概念候選）
疊起三軸的分界線：
- 軸一（JIT）：全自動結晶 / 要人確認的結晶
- 軸二（OS）：user mode / kernel mode
- 軸三（分散式）：AP / CP

**是同一條線——可逆性邊界。** 可逆的（搬移、索引、capture、draft、聯集 merge）全自動、user-space、AP；不可逆的（定案、改根公理、認母概念）要人、kernel-space、CP。三面獨立的鏡子（執行/資源/一致性）畫出同一邊界——按 [收斂](../concepts/收斂.md)「多條獨立推導＝真根」，這條邊界不是設計者硬切的，是真的。它其實是**延伸原則 5（可逆自動/不可逆要人）的 real-why 被照深**：原本 stated-why 只是「定案不可外包」，真 why 是它同時是 CP/AP 線、kernel/user 線。（母概念認定是人的事 → 待定案。）

## 下一步：一個元件，三軸交點
**為 `history/` event-log 造一個跨 repo 的「單調聯集合併 + 人閘控批准」協議。** 它同時關掉三軸缺口：OS 缺的並行控制、分散式缺的共識層、JIT 在多節點的 profile 共享。
- 你不在「強共識 vs 最終一致」二選一，按*資料類別*分流：事實/事件 AP 聯集，決定/原則 CP 人批。

**開放設計問題（待答 → 答了會變 design/experience）**：跨 repo 同步的單位是 **history event**（因果軌跡，天然單調好聯集）還是 **concept**（多方各自演化、難 merge）？
- 傾向主張：**只同步 event-log、concept 各 repo 從 log 本地重建**（＝CQRS：event 是共享真相，concept 是各自 read model）→ merge 永遠聯集、不撞 concept 命名衝突。
- 代價：同一份 events 在不同 repo 可能投影出不同 concept 組織。可接受？待定。

## 出口
