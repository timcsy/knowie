# migrate 跑起來會踩的問題（預測，趁早記）

> 2026-06-12 討論。draft，problem 清單——**演繹預測**的失敗模式（不是踩到才記）。
> 怎麼冒出來的：問「migrate 跑時還會有什麼問題」。是 [replay 模型](2026-06-12-migrate時間軸replay.md) / [遮罩 harness](2026-06-12-遮罩harness設計.md) / [架構視角](2026-06-12-migrate架構視角-ES-CQRS-DDD.md) 的下游風險盤點。

## 兩個最危險、且是「真實常態」的預設
現行設計默認了兩件事，而它們在真實世界大多**不成立**：
1. **乾淨線性 git 史**——但真實是亂 commit、且是 DAG（見問題 1）。
2. **從第一天就有 knowledge/**——但真實多是中途才採用 knowie（見問題 4）。
這兩個假設破，品質就崩。

## 🔴 高（幾乎一定踩）
1. **切片：真實 git 史亂、且是 DAG 不是直線**。多數專案無里程碑 tag / 無 conventional-commit scope / message 亂 → 切片無乾淨訊號。更糟：merge/branch/rebase → 「往前播的時間軸」不是直線；rebase 過的史失去真實時序。且切片若靠 LLM 判 → **切片邊界非決定性** → 非決定性從後門溜回（「決定性骨架」假設撐不住亂史）。
   - 🟩 **已有設計答案（寫進 migrate 2026-06-12）**：沿 `--first-parent` mainline、一 merge/PR=一片。DAG 不是問題是答案——mainline=domain events、branch 內部=technical events（DDD 投影）。切片來自 git 結構非 LLM 判讀 → 非決定性後門關掉。純線性無 merge → message-arc/固定窗 + 較粗顆粒。rebase 過的史更線性，只丟真實時序（可接受 proxy）。
2. **錯誤 cascade（循序的代價）**：第 N 片搬錯 → N+1 建在錯 base 上 → 錯誤往前複利；HITL 只准 read+additive、中途不能改 → 早期一錯後面全帶著走。單調性的反面。
   - 🟩 **已有設計答案**：分清「改記錄 ≠ 改過去」。單調性禁的只是「用後見之明重審判專案的過去決策」；修「轉錄錯誤」（sub-agent 記錯實際發生的事，如轉移指向 base 從沒記過的決策=judge 抓得到的 dead reference）不是撤回 → 准修、重 checkpoint。防火牆已在：每片 checkpoint + 每幾片 judge=cascade 偵測器。
3. **推論的 why 可能錯 / 根本沒有（非純 ES 的根源）**：IRL 從 commit 推 why 可能推錯，卻被當真記下（why 沒 [oracle](../concepts/why沒有oracle.md)、git 驗不了）；有些 commit 根本無可recover的 why → AI 要嘛編一個（更糟）、要嘛留洞。
   - 🟨 **半解（寫進 migrate）**：兩種聲音——stated-why（作者記的，權威）vs inferred-why（從 diff 推的，**標記成推測**），HITL 專挑推測去 jog。推不出 why → 留洞記成 episode、不給假 history 轉移。讓不確定可見、不洗成假自信。根治不了（why 沒 oracle 是根公理），只能誠實標記。
4. **knowledge/ 是後來才加的（超常見）**：中途才用 knowie → 早期 git 史無 knowledge/ →「讀當時 knowledge」對前半段是空的 → why 只能從 code/spec 推（最弱來源）。**多數真實採用都是這樣**。
   - 🟩 **已有設計答案（寫進 migrate）**：引入 **adoption commit=相位邊界**（引入 knowledge/ 的第一個 commit）。migrate 其實混了兩操作：結構遷移 vs 考古回填，接縫就是它。採用前=粗顆粒考古（why 弱、標 inferred）、採用後=細顆粒讀當時 knowledge。「讀當時 knowledge」降為 enrichment-when-present、per-slice graceful，不再是壞掉的全域假設。

## 🟡 中
5. **舊 base 有「無法從切片重建」的內容**：跨多 commit 慢慢磨的、或直接寫的（不綁單一 commit）→ 逐片增量碰不到 → cross-check 發現「舊有新沒有」卻補不回。需一條「無法重建的舊內容 → 直接搬 + 人確認」的路。
6. **成本/規模**：N 片 ×（sub-agent + 可能 worktree）+ 累積 base 越長越大、每片都餵 → 100+ commit 可能跑數小時、大量 token。實際可行性。
7. **轉移偵測 = 跨片實體匹配**：「這片改了先前決策」要先認出兩決策講同一件事（同一 schema/auth model）；措辭不同 → 匹配失敗 → 轉移漏掉、記成獨立新決策。entity-matching 很糊。
8. **中斷/resume 狀態**：structureVersion 只在最後 bump → 半完成 base 看似「沒遷移」→ 重跑從頭來（丟 checkpoint）。需「遷移進行中」標記 + knowledge.old/、worktree 殘留清理。
9. **遮罩洩漏會 propagate**：一片偷看 → 污染進累積 base → 後面每片都看到污染。一漏毒全部。

## 🟢 低（視專案）
10. **多貢獻者**：git 史交織多人 why；HITL 要 jog 哪個開發者？單一 coherent why 可能是多人衝突的 why。
11. **驗證結果**：人讀不動 50 檔去驗「why 準不準」→ ROI/信任問題。

## 已解 / 未解
- ✅ **NNN 編號**：循序＝單一寫者 → 不撞號（循序順手解）。
- 🟩 **1（亂史/DAG）、4（後加 knowledge）已有設計答案** 並寫進 migrate（first-parent 切片、adoption 相位邊界）。
- 🟨 **2（cascade）、3（推錯 why）有規則** 寫進 migrate（改記錄≠改過去 / 兩種聲音+不編造），但 3 根治不了（why 沒 oracle），靠標記+HITL。
- ❌ 未解：5–9 中段、10–11 視專案。

## 出口
- 預測清單，待真實驗哪些先爆。**1 和 4 不是邊緣 case，是常態 → 設計時就該正視**（別默認乾淨線性史 + 從頭有 knowledge/）。已正視：見上 🟩。
- 底線（四個是同一句話）：**別假設證據均勻乾淨，順著證據、標出強度**——1 按 git 結構切、4 顆粒度跟證據密度、3 why 按來源強度標、2 准修記錄只禁改過去。
