# 設計型 draft 無法預先測有效性（雞生蛋）

> 2026-06-06。draft，未定案。
> 怎麼冒出來的：想「先解決 draft 再開始用」時，點出有些 draft 沒實作就不知好不好用——而這是之後很常遇到的問題。

## 兩型 draft
- **經驗型**（踩坑教訓、討論結論）：使用中自然產生，蒸餾即可定案。好處理。
- **設計型**（機制提案：記憶框架 / tag / episodes / 程序自習得）：是**提案**，要實作 + 使用才能測有效性，而實作有成本。

## 設計型的雞生蛋
不實作 → 不知有效；不知有效 → 不敢實作；要定案（結晶為願景／原則）→ 需知有效。卡住。

## 破解
- **dogfood**：用 knowie 維護 knowie，就是**元層設計型 draft 的實驗場**。judge 試跑已實測「draft 整理機制」並揪出缺口（複習偵測沒機械化）——設計型 draft 真的被測了。
- **領域層**設計型（domain skill、實際開發流程）→ 要在**真實專案**才能測，dogfood 測不到。
- **分驗證方式**：能**內省判斷**的（邏輯一致性）→ 想清楚就定；**必須實測**的（AI 行為 / 人因 / 規模）→ 設計**最小實驗**，別全做。
- **借既有**：別自己造來測（記憶機制借 Letta/Hermes、skill 借 agentskills.io）。

## draft 該加的維度
除了「成熟度（複習信號）」，設計型 draft 該標**驗證方式**：內省 / dogfood / 真實專案 / 借既有。讓「它卡在哪、怎麼往前」清楚。

## 對「開始使用」的結論
設計型 draft **本來就無法預先測** → **更不該「解決它們才開始」**（那會永遠開始不了）。帶著它們開始，讓 dogfood + 真實使用逐步驗證。這正是 captured ≠ committed 的用法。
