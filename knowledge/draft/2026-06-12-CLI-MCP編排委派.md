# CLI/MCP 編排該委派到共用核心，不是各抄一份

> 2026-06-12 設計討論。draft，design 型。怎麼冒出來的：漂移審計發現 init/update 在 CLI（`commands/*.js`）和 MCP（`mcp-server.js` handler）各實作整套編排，已漂（update 漏 README/language、init 語言寫法分歧）。

## 問題
`init`/`update` 的編排（scaffold → templates → readmes → skills → detect → handshake → 寫 config）在**兩處各寫一遍**：
- CLI：`src/commands/init.js`、`src/commands/update.js`（含互動：語言選單、確認提示、工具多選）。
- MCP：`src/mcp-server.js` 的 `handleKnowieInit`/`handleKnowieUpdate`（非互動，吃參數）。

兩條平行 → 必漂（已驗：update 這次、init 的 DIVERGENCE 4 語言寫法）。逐個對齊撐不住，見 [experience「同一編排實作兩遍必漂移」]。

## 方向
**抽非互動核心，讓兩端共用**：
- `installReadmes()` 已是雛形——init 和 update（CLI+MCP）都呼叫同一支。把整套編排比照辦理。
- 形態：一個 `runInit(projectRoot, opts)` / `runUpdate(projectRoot, opts)` 純核心（不含 prompt、不含 console），回報 structured result。
  - **CLI** = 互動層（語言選單、confirm、工具多選 UI）決定 opts → 呼核心 → 把 result 印成人話。
  - **MCP** = 直接用參數組 opts → 呼核心 → 把 result 串成 report。
- 互動是**輸入來源**的差異（人選 vs 參數），不該讓**編排邏輯**分岔。

## 待決
- 互動點怎麼抽：語言/工具選擇是「決定 opts」的前置步，能乾淨切在核心之外嗎？（init 的工具多選 UI 最纏）。
- config 寫入語意要統一：CLI init 覆蓋 `config.language`、MCP 只在缺時設——核心該定一個意圖（建議：顯式傳入的 language 才覆蓋，否則保留）。
- structureVersion 落後警告：CLI update 有、MCP 沒有——移進核心 result，兩端都報。
