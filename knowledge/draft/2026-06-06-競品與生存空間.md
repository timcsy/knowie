# 競品三方定位（vs Letta / Hermes）與生存空間

> 2026-06-06，查證 Letta + Nous Research 的 Hermes Agent 後的對比。draft，未定案。
> 怎麼冒出來的：問「knowie 跟 Letta / Hermes Agent 距離多遠」，查完發現 knowie 的「記憶 agent」野心大半已被它們 production 覆蓋，但有一塊它們刻意留白。
> ✅ **已進 vision 核心想法**：FUSE 式 why 層定位（不競爭記憶 agent，當它們的 why 層）。本檔留作完整的競品分析與緣由。
> ⚠️ **2026-06-07 修正**：查證「LLM Wiki」（Karpathy 模式）後，發現「人在環是 knowie 獨特」這條**錯了**——見下方新增的第四方 + 修正。

## 三方是什麼
- **Letta**（MemGPT 系）：OS 式記憶平台，agent 自治、git-backed MemFS、skill learning。Production。
- **Hermes Agent**（Nous Research，2026/02，v0.9）：自主 agent + 多層記憶（MEMORY.md/USER.md 常駐 + SQLite FTS5 session search + 外部 provider）+ 自習得 skill（agentskills.io 開放標準）+ 全自動 + 多平台。Production。
- **knowie**：why 的協議，人在環、結構化（三視角/法典）、寄生 markdown。理論深、實作初級。

## 三方共通（已是業界共識）
markdown 記憶、多層、skill = 程序記憶、容量逼 curation、explicit + implicit 存、FTS5 + LLM 雙檢索。**這些 knowie 講的，它們都 production 了。**

## knowie 的距離
- **工程 / 能跑**：落後很遠（它們 production；knowie 協議 + 剛重寫未驗證 skill + draft 理論）。
- **程序自習得**：**Hermes 領先**（已做 + 開放標準化；knowie 還在 draft）。
- **可回憶 / 檢索**：概念打平、實作落後（它們有 FTS5 + LLM；knowie 的「可回憶單元」更深但紙上）。
- **結構化 why**：**knowie 獨特**——Letta/Hermes 都沒 why 層（Hermes：why「distilled into facts if critical」）。
- ~~**人在環**：knowie 獨特~~ → **2026-06-07 修正：不獨特**。LLM Wiki（Karpathy 模式）也強調「requires active human curation — not self-maintaining」。全自動只是 Letta/Hermes 那一系；curation 派早有人在。

## 第四方：LLM Wiki（Karpathy 模式）——最近的鄰居（2026-06-07 加）
查「連結語法該用哪種」時撞見的：**LLM Wiki 幾乎是 knowie 的獨立趨同雙胞胎**，且已多方實作（nvk/llm-wiki、vanillaflava、ar9av/obsidian-wiki），**Hermes 已收為 bundled skill**。

| LLM Wiki | knowie |
|---|---|
| wiki/concepts/ · entities/ · index.md · log.md · sources/ | concepts/ · episodes/ · Key Extensions 索引 · history/ |
| **wiki-crystallize**（session distillation） | **knowie-capture**（結晶/蒸餾）|
| **wiki-lint**（broken links/orphans/stale） | **knowie-judge**（死連結/孤兒/過時）|
| wiki-query（index→沿連結→必要時回 source） | knowie-next（三視角檢索）|
| wiki-config / ingest / integrate | knowie-init |

- 連用詞都撞（crystallize ↔ 結晶；lint ↔ judge）。**這驗證 knowie 的直覺是 zeitgeist，不是孤例。**
- **它偏好 `[[ ]]`**：因為 graph/backlink 是它的**檢索機制**（「every [[wikilink]] becomes an edge」），號稱 95% less token than RAG。成熟實作用 dual-link（`[[ ]]`+`[]()`）求相容。

## 最尖銳的結論（2026-06-07 修正）
knowie 的「記憶系統」野心，在「facts + skills + 全自動」上被 Hermes/Letta 輾過；「結構化 markdown + crystallize + lint + 人 curation」上被 **LLM Wiki** 覆蓋（連人在環都有）。**真正剩下、沒人做的是：結構化的 *why*（三視角 / 法典 / 因果軌跡）——不是「人在環」本身，是「人在環地策展 why 的結構」。** 這條要窄、要硬。
- 待答：把 knowie 蓋成「LLM Wiki + why 層」算不算更老實的定位？（它已驗證可行 + 有採用）vs 維持獨立協議。→ 見 [檢索機制-graph還是path](2026-06-07-檢索機制-graph還是path.md)。

## 戰略（生存空間）
1. **別比「記憶 agent」**——會被輾壓。
2. **當它們的「why 層」**——寄生在 Hermes / Letta / 任何 agent 之上，提供它們缺的「結構化 why + 人的定案主權」。這是「協議寄生平台」的終局。
3. **程序記憶別自己造**——用 agentskills.io 開放標準（Hermes 已採用）；knowie 專注 why。
4. **定位轉變**：knowie 不是「又一個記憶 agent」，是「facts/skills 之上、它們留白的 **why 層 / 人在環定案層**」。越早這樣定位，生存空間越清楚。

## 生存判斷（誰在做 / 潛力 / 未來需求）
- **誰在做**：「為共識結構化 why」有老實踐（ADR / Zettelkasten / design doc / 制度記憶），但純人工、靜態、易變墳場；agent memory（Letta / Hermes）有 AI 動態，但為 agent。**「為共識 + AI 維護動態 + 協議化」的交叉點目前空著。**
- **潛力**：順風＝AI 越自動，人越需要對齊 / 主權 / 不漂移共識；逆風＝中間地帶兩邊可能都不買單、逆「全自動」大勢、協議需採用。真實但不保證。
- **未來需求**：幾乎確定會在（專案級對齊 + 人的主權 + 可審計 why + 不漂移共識，隨 AI 自動化上升而上升）。但「需要這層」≠「需要 knowie」——可能被平台吸收。賭注在「**被吸收前卡位成協議**」。

## FUSE 式定位（對抗被吸收的核心防禦）
願景：knowie 像 **FUSE**——只提供**協議與運作方式**，不專注底層如何實現。
- knowie 定義「為共識而記的協議 + 運作方式」（三視角、capture / judge 行為、人在環定案、衰減 / 固化判準），**底層記憶實現可插拔**（markdown / SQLite / 向量庫 / 甚至 Letta、Hermes 的記憶後端）。
- knowie **不自己做記憶儲存 / 檢索的最佳化**（那是底層，讓平台 / 向量庫做）；可「掛載」在任何後端上。
- **對抗吸收**：平台要「吸收」knowie ＝ 要「實作 knowie 協議」——而協議被多方實作正是它的**成功**（像 FUSE 被各 OS 支援），不是被消滅。
- **挑戰**：FUSE 介面純機械、極穩極簡、可形式化；knowie 協議一半是結構（可形式化）、一半是行為判準（軟，靠 AI 判斷）。要像 FUSE 那樣可插拔 / 可被多方實作，必須持續「**軟介面硬化**」（結構承載語義）。這是北極星，還沒到。

## 對 vision 的影響（待落地）
這直接影響 knowie 的定位，將來可能該寫進 vision：**knowie = 寄生於 agent 記憶系統之上的 why 協議，不與記憶 agent 競爭**。現在留 draft，未定案。
