# 遮罩 harness 設計：讓 replay 真的往前播（執行層元件）

> 2026-06-12 設計。draft，design 型——先想清楚再建。
> 怎麼冒出來的：[migrate 時間軸 replay](2026-06-12-migrate時間軸replay.md) 的「執行層黑洞」——skill 寫往前播、但 AI 有 repo 存取權會偷看結局（hindsight），措辭擋不住。真解是**物理上拿掉未來**＝遮罩 harness。見 experience「有些 bug 是執行層的」。

## 為什麼是 harness、不是 skill
遮罩＝「**AI 看不到不該看的（未來/結局/舊最終態）**」。這是**對 AI 輸入的控制**，skill 措辭（對 AI 的指令）保證不了——指令攔不住自由 file read。所以遮罩屬**執行層**：一個**驅動器**控制每片 AI 能看到什麼。對應 [harness-loop](2026-06-12-harness-loop-workflow定位.md)：harness 提供遮罩+迴圈，knowie 提供「拿到一個遮罩切片該做什麼」的 skill。

## 核心迴圈
```
0. 切片：從 git 算出有序時間窗（里程碑 tag / spec 目錄出現 / conventional-commit scope）。決定性，算一次。
1. 把舊最終 knowledge/ 搬到一邊（knowledge.old/，replay 期間隱藏）。新 knowledge/ 從空開始。
for slice in 切片（順時間）:
   2. 建「遮罩 context」：只含 ≤ 此片 commit 的狀態（worktree@commit 或抽出該片檔案）。
   3. 餵 AI：遮罩 context + 目前累積的新 knowledge/（learning 側，可見）。
      AI 任務＝對這片做 IRL：推「為何當下這樣做」+ 分發（episode/history/concept/experience）。
   4. AI 輸出疊進新 knowledge/（累積）。
   5. checkpoint（commit）新 knowledge/。
6. 收尾：補 canon（Key Extensions/structureVersion）；拿 knowledge.old/ 做完整性 cross-check（新有沒有涵蓋舊、且 home 對）。
```

## 三種狀態的可見性（關鍵）
| | 狀態 | replay 期間 |
|---|---|---|
| **未來軌跡**（> 此片的 git） | 不可變 trajectory / replay buffer | **遮蔽**（偷看＝hindsight＝污染 why 推論） |
| **累積的新 knowledge/** | learning / policy（RL）；只增（CALM 單調） | **可見**（這就是「疊上去」） |
| **舊最終 knowledge/** | 「結局」的一種；跨多 commit 的最終策展態 | **搬走+隱藏**（在原地＝被偷看+被沿用，history/001 復發的一半原因）；只在收尾 cross-check |

## 最難的：confinement（怎麼真的擋住偷看）
「給過去的狀態」容易，「**擋住未來的狀態**」難——AI 有自由 file read 就能讀 main repo 的 HEAD。光開 worktree@commit 不夠（AI 仍可 `cd ..`/讀絕對路徑）。選項：
- **A. 沙箱化的 worktree**：per-slice AI 跑在一個 cwd＝worktree、且**檔案存取被限制在該 dir**（container/chroot/harness 中介所有 read）。最強，但重。
- **B. Harness 餵 constructed context**：harness 抽出該片材料、AI **只從給定 context 工作**（replay 期間沒有自由 Read/Bash）。AI 沒有自由 repo 存取＝沒得偷看。強，但 AI 進「受限模式」。
- **C. 自律 + worktree**（弱）：AI 自己開 worktree、自律不看別處——**已證無效**（就是現在的失敗）。
→ **confinement 是這個元件的真正價值**：不是「提供過去」，是「**否決未來**」。A/B 都行，C 不行。

## 分工（協議層 vs 執行層）
- **Harness（執行層）給**：切片排程（決定性）、遮罩+confinement、checkpoint、累積迴圈、舊 knowledge 搬遷+cross-check。
- **knowie skill（協議層）給**：拿到一個遮罩切片 → IRL 推 why + 分發（現行 migrate skill 的內容，**減掉**那些想用措辭做遮罩的部分）。
- → migrate skill 從「自己 replay」降成「**被 harness 呼叫的 per-slice 處理器**」。它不再假裝能遮罩（它不能）；它只把一片遮罩切片處理好。

## 它住哪（協議非平台 的張力）
harness 會**呼叫 LLM**（per slice），比現行 CLI（init/update，無 AI）重。三條路：
- **(a) `knowie migrate` 變 AI-invoking 驅動**：knowie 自己跑迴圈+confinement+呼叫 LLM。最完整，但 knowie 第一次「擁有一個會叫 AI 的 runtime」——偏離協議身分。
- **(b) FUSE 式：knowie 定「遮罩協議」**（切片 context 的格式 + per-slice skill 的介面），**任何 harness/loop-engine 實作**（含 CI、loop-engineering 工具）。最合協議非平台，但要有那種 harness 存在。
- **(c) 折衷**：knowie 定協議 + 附一個**參考驅動器**（thin script，confinement 用 worktree+sandbox）給常見情況（Claude Code），協議仍開放。
→ 傾向 **(c)**：協議是 spec、參考驅動器是一個實作、別人可換。守住「knowie 定協議+運作方式、底層可插拔」（FUSE）。

## 接既有（多面鏡子同一結構）
- **RL**：off-policy replay buffer（不可變軌跡，只讀）；confinement＝不讓 learner 偷看未來經驗。
- **event sourcing**：log 不可變、projection 重建——harness 就是「忠實 replay log 建 projection」的執行器。
- **CALM/單調**：新 base 只增（可見、安全）；未來遮蔽（不撤回）。
- **development atom / for 迴圈**：harness＝外圈迴圈，knowie＝迴圈外的 grounding，每片＝一個 atom。
- **兩相 + HITL**：這是 Phase 1（忠實 replay）；HITL 在此**只讀 + additive 補 why**（人在 learning 側）；Phase 2 策展是 harness 跑完之後。

## 開放問題 / 風險
- **confinement 強度 vs 成本**：A（沙箱 worktree）最穩但每片一次 checkout+sandbox，貴；B（餵 context）需要 AI 受限模式。要驗哪個夠用。
- **這還是 knowie 的 lane 嗎？** AI-invoking 驅動偏離「協議非平台」。(c) 用「協議+參考實作」緩解，但得守住「驅動器可換」。
- **切片邊界自動抓**：里程碑 tag / spec 目錄 / commit scope——哪個訊號穩？專案沒有清楚邊界時退化成什麼？
- **成本**：N 片 × （checkout + 一次 LLM pass）——比一次 pass 貴一個量級。粒度旋鈕壓成本。

## 出口（待建/待驗）
- 設計已具體，**待建**：(c) 的參考驅動器（切片→confined context→呼叫 skill→累積→checkpoint）+ confinement 機制（先試 worktree+受限 read）。
- **待驗**：建完拿 battle 跑——(1) 收斂性（二跑一致?）(2) 不再偷看（history/001 不復發?）(3) 轉移照長。**遮罩若對，收斂性八成同時解掉。**
- 先建**機械偵測檢查**（便宜、獨立於 harness）止血「沒偵測到」，再建 harness。
