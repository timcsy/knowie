# draft 的整理機制（何時整理、誰偵測誰執行）

> 2026-06-06 設計討論。draft，未定案。
> 怎麼冒出來的：問「何時該整理 draft」，從「capture 可以有 LRU 式偵測信號」的想法發展出來。

## 核心：capture 偵測，judge 執行
整理拆兩半，分給兩個 skill：
- **capture = 偵測（即時、便宜）**：每次寫 draft 時順手檢查信號（像 LRU 在每次存取時更新統計）。capture 是 draft 的存取點，所以是放偵測的最佳位置。
- **judge = 執行（批次、慎重）**：真正的固化／放手是定案級（固化會寫進長期層），留給 judge 批次處理。

→ 這精修了「judge 巡 draft」：偵測前移到 capture（每次都做），執行留 judge（離線批次）。

## 三個觸發信號（capture 偵測）
1. **容量壓力**（LRU 經典）：draft 超過軟上限 → 該整理。**不是定時清，是滿了才清**（壓力驅動 > 時間驅動）。
2. **複習信號**：某主題第 N 次出現（如「記憶類比」→「記憶系統框架」）→ 該固化。
3. **衰減信號**：某主題太久沒碰（git recency）→ 該放手。

## 比 LRU 多一維：三元，不是二元
LRU 只「留 or 丟」。draft 的常被碰項**不是留著，是畢業升級到長期記憶**。所以是三元，由 recency × frequency 決定：

| 信號 | 處置 |
|---|---|
| 高頻（反覆碰） | **固化** → 升級進三檔／concepts（複習效應，畢業） |
| 低 recency ＋ 低頻 | **放手** → 淡出／刪 |
| 中間 | **繼續孵化** → 留 draft |

draft 因此更像「記憶固化」（海馬迴 replay → 固化到新皮質），不是純 cache 淘汰——LRU 沒有「畢業」出口，draft 有。

## 何時整理：信號驅動，非定時
capture 每次寫入時偵測「容量／複習／衰減」，達閾值就**標記**該主題（該固化／該放手）；judge 批次巡檢時**執行**標記。整理 = 持續偵測 ＋ 批次執行，不是定期大掃除。

## 對 skill 的影響（待落地）
- **capture** 加職責：寫入時順手做 LRU 式偵測 ＋ 標記。
- **judge** 的 draft 巡檢：執行 capture 標記的固化／放手（已在 judge 的 draft 巡檢雛形裡，可深化）。
