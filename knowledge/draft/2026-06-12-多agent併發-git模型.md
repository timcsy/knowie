# 多 agent 同時用 knowie：併發 = git 的併發

> 2026-06-12 設計討論。draft，未定案（設計/定位題，大半未建）。
> 怎麼冒出來的：問「多個 agent 可以同時用 knowie 嗎」。是 [跨 session/跨廠商 handover](2026-06-12-跨session跨廠商handover.md) 的**空間軸並行版**（handover＝時間軸接力）。
> 核心命題：**knowie 沒發明併發模型，它繼承 git 的，再疊兩樣它本來就有的東西（judge、人定案）。**

## 讀 vs 寫
- **讀**：無腦 yes（唯讀無衝突，N agent 共享同一份 why＝設計目標）。
- **寫**：＝ git 的分散式合併。knowie 幾個**為別的理由做的選擇**剛好讓併發友善：
  - `分發非傾倒` ＝ 天然**分片**：一概念/一 episode/一決策一檔 → 不同主題寫不同檔 → 不衝突。
  - append-heavy 層（history/episodes/draft 開新檔）＝ 結構上的 **CRDT（grow-only set）**，無衝突可交換合併。
  - 延伸原則 8（`[]()` 不 `[[ ]]`、graph 衍生不儲存）＝ **沒有中央索引要搶**。

## 三個理論視角（每個對上一個既有設計選擇）

### 工程 / 分散式系統
- **樂觀併發**：不鎖、自由寫、合併才解衝突；前提「衝突稀有」由分發分片滿足。
- **CRDT**：append 層 ＝ grow-only set，天生無衝突。
- **CAP → AP**：可用優先，**最終一致**（git partition-first）。
- **Anti-entropy / read-repair**：**judge 就是 anti-entropy**；`收斂` ＝ convergence。
- **Leader 序列化**：**人 ＝ 不可逆寫的單一權威**（root 原則/定案只有人 commit）。
- **缺口：write skew（寫偏斜）**：兩 agent 各讀一致狀態、寫不重疊、合起來違反不變量（git 不報衝突，但語意不一致）。knowie **無 serializable isolation** → 對 write skew 脆弱；judge 是**延遲偵測器**（deferred constraint check），中間有不一致窗口。這是「git 合併文字 vs knowie 要語意一致」的精確落差。

### Pipeline
- **WAW hazard** ＝ 同檔同時寫；分發分片消掉多數 hazard，殘留＝三檔/Key Extensions 熱點。
- **不 stall，resolve-after**：不鎖、先寫後修，像**投機執行 + 誤判回滾**（git revert）。
- **批次 drain**：judge ＝ pipeline 的 commit/drain 階段，**批次**對帳（＝`蒸餾`「整理=批次結晶」＝ write-behind / deferred commit）。別逐寫 reconcile。

### 作業系統（最深）
- **Share-nothing 預設**：分發導向私有檔 → 避開 critical section；殘留共享可變狀態（三檔）才是不可避免的 critical section。**協調只放無法避免處**。
- **保護環（同構，非比喻）**：**延伸原則 5「可逆自動 / 不可逆要人」＝ ring 3 vs ring 0**。agent 跑 user mode（可逆、被 git 可回滾性 sandbox）；不可逆寫要 **trap 到人這個 kernel** 序列化。
- **Release consistency**：寫到**同步點**才可見（pull＝acquire / commit-push＝release）＝鬆弛記憶模型的 DSM。
- **git commit ＝ 交易 + WAL**：commit 原子；多檔分發塞一個 commit 就有原子性；commit log ＝ 序列化排程。
- **無鎖 → 無 deadlock**（代價：事後解衝突換事前防死鎖）。

## 綜合架構（一句話）
**share-nothing 分片（CRDT 葉子）+ git 樂觀/最終一致/MVCC + judge 當 anti-entropy + 人當不可逆寫的保護環序列化點。**
各層按**一致性需求**自然分裂：
- **冷/大/分片的葉子**（history/episodes/draft）＝ CRDT、AP、零協調。
- **熱/小/共享的介面**（三檔/Key Extensions）＝ critical section；其中不可逆部分用**人序列化 ＝ 強一致核心**。
→ 好系統鐵律：**強一致只留給小核心，其餘放生最終一致**。knowie 為可攜/分發/衍生 graph 的選擇，掉出一個教科書級併發架構。

## 最佳實踐
每 agent 一個 git worktree/branch、**按主題分片** → 各寫各的 → merge → 跑一次 judge 對帳語意。

## 刻意的非目標
**不做協調器**（＝runtime＝平台，背叛 [協議非平台](../concepts/協議非平台.md)）。併發上限＝git 的上限——特性非 bug：可被任何懂 git 的人推理，不綁平台。同檔熱點的同時寫，fallback 到 git 衝突 + 人解。
