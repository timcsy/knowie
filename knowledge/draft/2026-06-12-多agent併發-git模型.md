# 多 agent 同時用 knowie：併發 = git 的併發

> 2026-06-12 設計討論。draft，未定案（設計/定位題，大半未建）。
> 怎麼冒出來的：問「多個 agent 可以同時用 knowie 嗎」。是 [跨 session/跨廠商 handover](2026-06-12-跨session跨廠商handover.md) 的**空間軸並行版**（handover＝時間軸接力）。
> 核心命題：**knowie 沒發明併發模型，它繼承 git 的，再疊兩樣它本來就有的東西（judge、人定案）。**

## 讀 vs 寫
- **讀**：無腦 yes（唯讀無衝突，N agent 共享同一份 why＝設計目標）。
- **寫**：＝ git 的分散式合併。knowie 幾個**為別的理由做的選擇**剛好讓併發友善：
  - `分發非傾倒` ＝ 天然**分片**：一概念/一 episode/一決策一檔 → 不同主題寫不同檔 → 不衝突。
  - append-heavy 層（history/episodes/draft 開新檔）＝ 結構上的 **CRDT（grow-only set）**，無衝突可交換合併。
  - 延伸原則 8（`[]()` 不 `[[ ]]`、graph 衍生不儲存）＝ **沒有中央索引要搶**。

## 三個理論視角（每個對上一個既有設計選擇）

### 工程 / 分散式系統
- **樂觀併發**：不鎖、自由寫、合併才解衝突；前提「衝突稀有」由分發分片滿足。
- **CRDT**：append 層 ＝ grow-only set，天生無衝突。
- **CAP → AP**：可用優先，**最終一致**（git partition-first）。
- **Anti-entropy / read-repair**：**judge 就是 anti-entropy**；`收斂` ＝ convergence。
- **Leader 序列化**：**人 ＝ 不可逆寫的單一權威**（root 原則/定案只有人 commit）。
- **缺口：write skew（寫偏斜）**：兩 agent 各讀一致狀態、寫不重疊、合起來違反不變量（git 不報衝突，但語意不一致）。knowie **無 serializable isolation** → 對 write skew 脆弱；judge 是**延遲偵測器**（deferred constraint check），中間有不一致窗口。這是「git 合併文字 vs knowie 要語意一致」的精確落差。

### Pipeline
- **WAW hazard** ＝ 同檔同時寫；分發分片消掉多數 hazard，殘留＝三檔/Key Extensions 熱點。
- **不 stall，resolve-after**：不鎖、先寫後修，像**投機執行 + 誤判回滾**（git revert）。
- **批次 drain**：judge ＝ pipeline 的 commit/drain 階段，**批次**對帳（＝`蒸餾`「整理=批次結晶」＝ write-behind / deferred commit）。別逐寫 reconcile。

### 作業系統（最深）
- **Share-nothing 預設**：分發導向私有檔 → 避開 critical section；殘留共享可變狀態（三檔）才是不可避免的 critical section。**協調只放無法避免處**。
- **保護環（同構，非比喻）**：**延伸原則 5「可逆自動 / 不可逆要人」＝ ring 3 vs ring 0**。agent 跑 user mode（可逆、被 git 可回滾性 sandbox）；不可逆寫要 **trap 到人這個 kernel** 序列化。
- **Release consistency**：寫到**同步點**才可見（pull＝acquire / commit-push＝release）＝鬆弛記憶模型的 DSM。
- **git commit ＝ 交易 + WAL**：commit 原子；多檔分發塞一個 commit 就有原子性；commit log ＝ 序列化排程。
- **無鎖 → 無 deadlock**（代價：事後解衝突換事前防死鎖）。

## 綜合架構（一句話）
**share-nothing 分片（CRDT 葉子）+ git 樂觀/最終一致/MVCC + judge 當 anti-entropy + 人當不可逆寫的保護環序列化點。**
各層按**一致性需求**自然分裂：
- **冷/大/分片的葉子**（history/episodes/draft）＝ CRDT、AP、零協調。
- **熱/小/共享的介面**（三檔/Key Extensions）＝ critical section；其中不可逆部分用**人序列化 ＝ 強一致核心**。
→ 好系統鐵律：**強一致只留給小核心，其餘放生最終一致**。knowie 為可攜/分發/衍生 graph 的選擇，掉出一個教科書級併發架構。

## 最佳實踐
每 agent 一個 git worktree/branch、**按主題分片** → 各寫各的 → merge → 跑一次 judge 對帳語意。

## 往下挖：同構是預測性的，不是裝飾（2026-06-12 續挖）
若 knowie ≈ 分散式系統成立，這個透鏡該能**預測** knowie 的真實問題。往前推挖到三樣：

### (a) CALM 定理 — 把延伸原則 6 重新接地到「協調自由」
**CALM（Consistency As Logical Monotonicity）**：一個分散式程式能**無協調**地保持一致 ⟺ **它是單調的**（只增不減、不撤回）。
- append 層（history/episodes/draft、加概念）＝單調＝無協調（＝上面的 CRDT）。
- **轉折**：修訂一條原則本該非單調（撤回＝要協調）。但**延伸原則 6「記錄轉移、不刪、標 superseded、連到新的」把非單調的「更新」變成單調的「append」**（加一筆轉移，不原地改掉）。
- → **延伸原則 6 不只是可審計，它祕密在做「單調化」→ 把更多系統推進 CALM 的無協調區。** knowie 不知 CALM 卻本能應用了它。
- CALM 反面也準：**協調（人序列化）需要的地方，恰恰是單調性破掉處**——「現在什麼為真」的當前狀態介面（principles/vision 當前文字）。**這不是設計選擇，是定理逼出的必然。**

### (b) knowie 其實是 Event Sourcing + CQRS — 重新解釋「可回溯」
- `history/` ＝ append-only **事件日誌**；三檔 ＝ **投影／物化讀模型**（CQRS read model）；`蒸餾` ＝ **log compaction**（LSM/Kafka 壓實／snapshot）；**根公理二「可大致回溯」＝ event sourcing 的 replay**。`記憶系統`「三視角從底層蒸餾析出、析出後自立有權威」字面就是 CQRS 投影。
- **CALM 逼出的關鍵選擇**：純 event sourcing 的投影是**自動推導**（無協調、可重建、但無人的權威點）；**knowie 選擇投影被人 anoint 成權威**（人定案）→ anoint 處要協調，但保住定案主權。
- → **CALM 證明不能兩者兼得**（可變的權威當前狀態 vs 協調自由）。**knowie 選了權威（為共識），所以核心的人協調不是缺功能，是「選人權威」的結構性代價——除非放棄使命，否則不可逃。** 這把「核心為何不能 CRDT」從「還沒做」升級成「不可能」。

## ⚠️ 真實瑕疵（獨立標出）：`history/NNN` 是分散式計數器
> 這不是哲學，是個**會在多 agent 真實爆的 bug**。**已收斂出解（見下「解」）**——方向定了，落地留到併發/migrate 真正上場。

`history/001、002、003…` 是**全域遞增序列**，而全域計數器**需要單一寫者**。
- **多 agent 並行下會壞**：兩個 agent 各開 branch、各要記一個決定 → **都挑到 `009`** → merge 撞號（或靜默蓋掉）。**跟「naive auto-increment 主鍵在分片 DB 爆掉」同一個 bug。**
- **NNN 假設了單一寫者**，與「多 agent 併發」直接矛盾。

### 解（收斂）：NNN 降級成「局部邏輯時鐘」，權威順序交給因果偏序
根因：**NNN 把本質是偏序（DAG）的東西壓成假的全序。** 獨立的決定（如「連結用 `[]()`」vs「crystallize 改名」）**本來就無因果順序**；NNN 的 `<` 反映「我剛好先寫哪個」，非因果必然。真有序的只有「決定 ↔ 取代它的決定」（supersedes 邊）。

收斂解（＝重新詮釋，幾乎不改機制）：
- **NNN ＝ 局部邏輯時鐘（Lamport clock）**：各 agent 在自己 branch `max+1`，**純局部、零協調**。**撞號是預期且無害的**——全名仍唯一（`009-foo` ≠ `009-bar`），連結照解；merge 後下一個接 `max+1` 續上，撞號只是孤立 blip。**撞號 = 並行（concurrent）的誠實訊號**，不是 bug。
- **權威因果順序 ＝ 偏序**：supersedes/(新增)follows 連結 + git DAG（happens-before）。
- **NNN 的角色降為**：(a) 人類可讀標籤；(b) **瀏覽排序提示**——`ls history/` 用 NNN 當 sort key 白送一個 linearization 給人看。要確定線性化就 tie-break `(NNN, agent/slug)`＝Lamport 的合法投影。
- **caveat**：NNN-sort 是**近似**線性化，跨 branch 可能與因果矛盾（A 被 B 取代，但局部編號 A=012 > B=009）。規則：**瀏覽看 NNN（夠用）；要正確因果，跟連結走（連結贏）**。同 Lamport clock 的已知限制（時間戳不能跨 process 判因果）。

三重對齊（這個落點不是 ad hoc）：**Lamport/CALM**（局部時鐘+因果調和＝經典無協調招；DAG 單調＝CRDT）／**延伸原則 8**（順序衍生不承載）／**根公理一**（一份因果 DAG，多個合法線性化；NNN 只是凍結的一個 view）。

→ 最佳解：**NNN 不必死，降級成局部時鐘即可**——保住可讀性 + `ls` 排序便利，同時拿到協調自由。比「merge 重編號」（要工）和「換 ULID」（丟可讀性）都省。

## 刻意的非目標
**不做協調器**（＝runtime＝平台，背叛 [協議非平台](../concepts/協議非平台.md)）。併發上限＝git 的上限——特性非 bug：可被任何懂 git 的人推理，不綁平台。同檔熱點的同時寫，fallback 到 git 衝突 + 人解。
