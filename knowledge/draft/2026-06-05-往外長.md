# knowie 往單專案之外長

> 2026-06-05 設計討論。draft，未定案。
> ✅ **已進 vision roadmap**：「版本偵測 + 遷移提示」（協議版本那段）。跨專案／個人方法論層結構未定，留此。
> 🔨 **版本偵測已實作（2026-06-07，roadmap 在建脈絡）**：
> - 用 **`structureVersion`**（新欄位，與 tool `version` 脫鉤）——只在**結構真的改**時才升（`STRUCTURE_VERSION='2'`），所以小改不誤觸發遷移（回應「之後有些微變動仍可使用」）。
> - 偵測**不只信版號**，而是**比對當前正典訊號**（`_core` Structure + `.templates/`）——未來結構小改自動涵蓋，不用改 skill。
> - 抓到並修掉一個 bug：`init`/`update`（CLI）原本無條件 `config.version=VERSION`，舊結構專案一更新就被升版＝**靜默 breaking**。修法：CLI 只升 tool version、**不碰 structureVersion**；落後就印 `structureBehind` 警告；structureVersion 只由 `/knowie-migrate` skill 遷移後才升。
> - 完成→兌現出列時，這段「為何兩個版號」的決策該蒸餾進 history。
> 🔨 **git 時光機（2026-06-12 加進 migrate skill）**：migrate 原本只比對「當前狀態 vs canon」，沒有「當時的畫面」。加一段判斷指引——**re-home 有歧義時用 git 還原當時脈絡**（`git log`+commit message ＝ knowledge 演化／`git show <commit>:path` ＝ 當時的 code/spec／順序 ＝ 轉移）。
> - **關鍵洞見**：**git ＝ 原始因果基底；`history/` ＝ 從它蒸餾的 why 軌跡**。所以舊 base（沒有 `history/` 層）的 **git log 就是可蒸餾成 `history/` 的因果軌跡**——migrate 從「搬資料夾」升級成「**還原失去的因果維度**」。操作化了根公理二「記憶可大致回溯」（git 是讓回溯可能的基底），並用 commit 指針補「可回憶單元的 how 腿」。
> - **四護欄**：git 是判斷的脈絡、**不是拿來 import the what**（仍只寫 why+指針，不複製 code）；**有界**（歧義時才翻，非考古坑）；**優雅降級**（沒 git → 退回當前狀態比對）；**人仍確認**。
> - 也能延伸餵 judge（對帳 why↔code 看何時分岔）/ capture（commit message ＝ why 來源），但 migrate 是最強 case。完成→兌現時這段該蒸餾進 history。

knowie 目前的設計都在「單一專案」內。往外長有兩個方向還沒想清楚：

- **跨專案 / 個人方法論層**：十幾個專案通用的 why = 個人方法論，是「概念收斂」再往上一階（專案之上那一層）。結構未定。
- **協議版本 / 遷移**（2026-06）：改了結構（research/design/history → draft/history/concepts）後，十幾個既有專案是舊的。
  - **決定：暫不做向後相容。** 理由：(1) 目前單人使用，無外部生態要保護；(2) 多數專案還沒累積到有子資料夾要遷（三檔 介面 沒變，遷移 ≈ 0）；(3) 向後相容的價值 ∝ 使用者數 × 舊資料量，現在兩者都 ≈ 0，成本卻 > 0。
  - **趁年輕做乾淨的 breaking change**——這是黃金窗口；向後相容是「老協議」的約束（POSIX/FUSE），knowie 還年輕，趁打掉重練便宜時改對。
  - **但保留版本偵測（≠ 相容）**：`.knowie.json` 的 `version` 讓 update 能「認出舊結構 + 提示手動遷移」，避免**靜默** breaking。breaking 不可怕，靜默 breaking 才可怕。
  - **遷移執行**：少數累積多的專案，作者手動處理；多數未累積的 ≈ 0。
  - **修訂自**：本條原寫「向後相容是硬約束」，被作者的真實處境（單人、未累積）校正——理論被經驗修正的一例。
