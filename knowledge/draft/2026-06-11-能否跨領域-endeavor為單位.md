# knowie 能跨出軟體嗎？單位是 endeavor，不是 topic

> 2026-06-11 設計討論。draft，未定案（problem 型）。
> 怎麼冒出來的：問「不是專案開發的話，三視角還有用嗎 / knowie 能當別的領域的知識庫嗎」。是 [往外長](2026-06-05-往外長.md) 的更上一階——跨**領域**（不只跨專案）。

## 分層：通用核心 vs 軟體封裝
- **通用核心**：三視角（principles/vision/experience）+ 記憶動態（capture/consolidate/judge/draft/concepts/episodes/skills/history）。三視角是 Frame 的知識論（他用在倫理/神學，不是軟體）→ 本就通用。
- **軟體封裝**（不直接轉）：根公理二「code 是 what 的真相」（非軟體領域連 what 都沒 oracle，那個乾淨的 what/why 分工糊掉）、next→spec→code→judge 迴路的 spec/code 具體、SDD 夥伴定位。
- → 換領域 = 保留核心、換掉軟體層。
- **精煉：軟體封裝其實是「artifact 封裝」**。code 只是 **artifact（產物＝what）** 的一種；照片、手稿、論文都是 artifact。knowie 一律「**持 why、指向 artifact**」（同 episode 連 spec、skill 連執行體）。換領域＝換「artifact 是什麼」（code→照片→手稿），核心（持 why、指過去）不變。

## 真正的分界：endeavor vs topic-web（不是軟體 vs 非軟體）
| | **endeavor**（有 why 的志業） | **topic-web**（關於世界的知識） |
|---|---|---|
| 例 | 寫書、經營團隊、研究計畫、「成為廚師」 | 維基、所有筆記、「我所知的法國史」 |
| 單一 vision/方向 | 有 | 沒有 |
| 三視角合身 | **合** | **不合**（沒願景、沒做中學的經驗） |
| 該用 | **knowie** | LLM Wiki / Zettelkasten |

同主題看怎麼框：「我**所知的**烹飪」＝topic-web（不合）；「我**成為廚師這條路**」＝endeavor（合）。

## 為何 knowie 需要 endeavor
整套機具都預設「正在做、往某處去」：vision＝往哪走、experience＝做中學、next＝規劃下一步、judge＝對帳產物、使命「為共識而記」＝對「我們在做什麼」的共識。**拿掉 endeavor，這些全沒東西可咬** → 只剩 concepts/episodes ＝ 一個沒三視角的 topic-web ＝ 退化成 LLM Wiki，特色全失。

## 結論 + 跟 LLM Wiki 的關係
- **能**跨出軟體——任何 **endeavor**（書/團隊/研究/創作/個人志業）都行；軟體只是已驗證的那一種。
- **不該**當通用 topic-web/second-brain（那是 LLM Wiki 地盤）。
- 釐清：knowie 跟 LLM Wiki **不是競品，是不同單位**——LLM Wiki 組織「主題知識」，knowie 組織「一個志業的 why」。（補強 [競品與生存空間](2026-06-06-競品與生存空間.md)。）

## 具體測試：照片（驗證邊界）
「knowie 能當照片知識庫、用 markdown 索引照片嗎？」
- **「索引照片收藏」** = 編目一個收藏 = **topic-web + 編目 what** → ❌ 不是 knowie（沒單一 vision、沒做中學的 experience；照片是 what，knowie 不管 what）。這是媒體資產管理 / LLM-Wiki 式索引的地盤。用三視角結構塞照片 metadata＝用錯形狀。
- **「攝影實踐 / 一本攝影集 / 紀錄片專案」** = endeavor → ✅ knowie 持那個志業的 **why**（信念/方向/教訓），**指向**照片；**照片本身是 artifact（what），連過去不收進來**（同 code）。
→ 兩格各歸位，乾淨驗證了 endeavor vs topic-web，並逼出上面「artifact 封裝」的精煉（照片＝code 在這領域的對應物）。
