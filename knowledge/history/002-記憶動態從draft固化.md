# 002：記憶動態從「擺在 draft」到「固化進 skills」

> 一條設計轉移的因果記錄。

## 轉移
- **A（曾做法）**：記憶動態（兩條輸入線 / 認識鏈 / 回流分發 / 分層整理）擺在 `draft/記憶動態`，等真實使用驗證才固化。
- **B（現採用）**：磨穩的機制固化進 `_core` / capture / judge + episodes 進結構；draft 只留還沒想清楚的部分。

## 為什麼變
原本以「captured ≠ committed」為由不動手——但那條的本意是「想清楚才 commit」，不是「永不 commit」。記憶動態這套已反覆討論磨穩，卻拿半成品去測真實專案，反而讓別的專案 knowledge 也變半成品。保守在這裡不是穩健，是裹足不前。

修正後的判準：**根公理（不可逆）該等；已磨穩的機制（可逆、可改）該主動固化**，讓產品先完整，再拿去測 dogfood 摸不到的 domain 層。

## 旁證
這正是延伸原則「可逆的自動化，不可逆的要人」的應用——機制固化可逆（隨時能改），所以該動手；根公理候選仍留 draft（不可逆，需慎）。

## 狀態
✅ 已固化（B）。教訓「成熟的設計該主動固化」已回流 experience。
