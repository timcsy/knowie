# 003：連結慣例定 `[]()`、檢索外包後端（why 層不自建 graph-RAG）

> 一條設計轉移的因果記錄。

## 決策
- 連結語法一律 **`[]()`**（標準 markdown）；不採 `[[ ]]`、不採 dual-link。
- 雙向/graph **由掃描衍生**（judge 職責），或物化少數關鍵節點。
- 大規模 graph 檢索 / RAG 替代 **不自建**，外包可掛載後端；adapter 從 `[]()` 結構衍生 graph。
- 定位：knowie = 掛在 LLM Wiki / agent memory 之上的 **why 層**，不比 wiki。

## 被否決的選項（純金 why）
- **`[[ ]]`（wikilink）**：在 GitHub/純 markdown 不渲染（第三方看到壞字）、要 resolver（= 假設平台，逆延伸原則 7）、AI 跟連結要多一次搜尋。其唯一真優勢（Obsidian raw 瀏覽即時 backlink）對「讀者主要是 AI」用不到。
- **dual-link（兩個都寫）**：寫兩份 = 冗餘，逆「砍冗餘/單一真相源」。除非 graph 邊真拿去檢索，否則是 theater。
- **自建 graph-RAG 檢索**：那是 *what-retrieval* 最佳化，是 Letta/Hermes/LLM Wiki 已贏的競技場；knowie 既定位「不比記憶 agent、當 why 層」，自建就走回剛避開的場子。

## 三個「該選 graph」的理由為何都倒了
1. **token 效率 / RAG 替代**：是 what，外包後端即得（把 knowie 的 md 餵進 LLM Wiki 就有，且因多了 why-邊更好）。
2. **graph 要靠 `[[ ]]`**：不必——graph 是衍生視圖，由 adapter 從 `[]()` 生（一份真相多投影＝根公理一）。
3. **雙向要靠 `[[ ]]`**：不必——雙向是「對正向連結的衍生反向索引」，grep 即得（judge 死連結偵測同一掃描順手吐）；或物化反向邊（knowie 現已在做）。

## 旁證
查證「LLM Wiki」（Karpathy 模式，多方實作、Hermes 已收）後確認：它偏 `[[ ]]` 正因為 graph 是它的檢索機制——而那恰是 knowie 決定不下場比的東西。見 [draft/競品與生存空間](../draft/2026-06-06-競品與生存空間.md)。

## 連帶
