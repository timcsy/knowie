# 三路並行漂移審計：從一個缺 README 的 bug 挖出整類問題

> 2026-06-12。怎麼冒出來的：battle 用 0.6.8 重跑，使用者發現 history/draft 檔名格式不對；追到根因是 `knowie update` 從不裝 subdir README，使用者再問「是否還有類似『改一邊忘另一邊』的問題？沒有唯一真實」。

## 場景
追 battle 檔名 bug 時，使用者沒停在「修這一個」，而是退一步問：**這種「改一邊忘另一邊」的病，還有沒有別的？** 我並行派了三個 Explore agent，各掃一軸：
1. CLI（`commands/*.js`）vs MCP（`mcp-server.js`）同操作雙實作的行為差異。
2. 硬寫死的清單/常數重複（`CORE_FILES`/`SUBDIRS`/`SUBDIR_READMES`/版本）。
3. 約定在多處 prose 重述的漂移（skills 之間、_core vs README）。

## 撞到的坑
- **agent 會讀到舊版**：審計 agent 報「MCP update 沒呼叫 installReadmes」「不裝新工具 handshake」兩條 critical，**驗證後都是 false alarm**——前者讀到我剛改前的快照、後者看漏了 newTools 先被 fold 進 existingTools。教訓：**子 agent 的 critical 結論一定要自己 grep 驗一遍再信**，並行審計省時但不省驗證。
- **真貨**：`CORE_FILES` 在 mcp-server 硬寫死兩次、`SUBDIR_READMES` 沒從 SUBDIRS 導出、_core 語言規則漏列 draft——都是同一病灶（唯一真實缺席）。

## 收穫
教訓本身（重複必漂移、解法是單一 source + 委派）回流到 [experience](../experience.md)；CLI/MCP 編排委派的設計 → [draft](../draft/2026-06-12-CLI-MCP編排委派.md)。**並行 fan-out 審計 + 逐條自驗**這個流程本身值得記——一個 bug 不只修它，退一步問「這是哪一類」，用多 agent 把同類掃乾淨。

- 三 agent 審計報告（本 session 內）。
