## 根本原則

> 🟡 待逐條確認定案。根本原則改動率應趨近零。

### 根本原則一：一個概念，多個投影
<!-- 推導自：無（這是根公理本身）-->
同一個真理可以、且應該被多個視角看；視角是 perspectives-on-the-whole，不是 parts（源自 John Frame 的三視角，但承重牆是經驗，不是 Frame——見 [[history/001-從自動到人在環]] 旁的 experience 教訓）。

- **意義**：知識不靠「分類到單一位置」組織，靠「概念 + 它在各視角的投影」組織。
- **如何應用**：整理時保持概念完整，把它投影到各視角，而不是把它切碎分進不同資料夾。

### 根本原則二：knowie 管 why，不管 what
<!-- 推導自：根本原則一 + 「code 是 what 的 ground truth」這個事實 -->
code 完整記錄 what 且是 what 的真相；knowie 只守 code 記不住、又不會自動報錯的 why。

- **意義**：重複記錄 what 是冗餘且必然與 code 不同步。
- **如何應用**：三檔記原則、路線、教訓（why），不記 API 文件、架構圖（what — 讓 code 自己說）。

## 延伸原則

<!-- 每條標明推導自哪條根本原則。可演化，但不得牴觸根本原則。 -->

### 1. 整理是投影，不是分割
推導自：根本原則一。把一個概念切碎分進不同類型資料夾，就違反了根公理。按概念組織，不按類型。

### 2. 捕捉，不是發明
推導自：根本原則一。概念是長出來的、名字是認出的、整理是事後的——皆 emergent。不預先設計概念，等它從具體中結晶。

### 3. 自我維護必須收斂
推導自：使命（why 要可信）。整理後再跑一次應近乎 no-op（冪等）；概念向少數根收斂。發散的自我維護 = 自動製造熵。

### 4. 事後可審計 > 事前確認
推導自：使命 + 協議定位。一切在 git 裡，AI 自動跑、人事後審/回滾。「自動」該等於「自動 + 留痕」，不是黑箱。

### 5. 可逆的自動化，不可逆的要人
推導自：使命。搬運/索引/修剪全自動；定案、修訂根本原則、認定母概念是不可外包的人類判斷。

### 6. 記錄轉移，不只記錄狀態
推導自：根本原則二（why 包含「為何變」）。改動留「為什麼變」；舊的標 superseded 並連到新的，不刪。

### 7. 協議，不是平台
推導自：使命（任何 AI 可用）。knowie 不擁有 runtime，只定義「結構 + 行為約定」，寄生在「AI 會讀寫 markdown」這個普世能力上。簡單與工具中立是可攜性的前提，不是美學。
