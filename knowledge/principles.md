## 根本原則

> 根本原則改動率應趨近零。
> 2026-06-07 **首次經人定案修訂**（見 [history/004](history/004-補使命-根公理二鬆綁.md)）：補上**使命**（填掉延伸原則 3/4/5/7 一直引用卻缺的洞）；根公理二由「管 why，不管 what」鬆綁為「記憶可大致回溯」。

### 使命：為共識而記
<!-- telos：回答「為何記、為誰記」。延伸原則 3/4/5/7 推導自此。 -->
knowie 為「人 + AI 的共識」而記憶——讓雙方對齊、人保有定案主權、可回溯「為什麼」。這是 knowie 存在的目的；其餘根公理與延伸原則都服務它。

- **意義**：記憶不是為了 agent 變強（Letta / Hermes 那條根），是為了人與 AI 對同一套 why 達成、維持、可追溯的共識。
- **如何應用**：設計取捨衝突時，問「哪個更服務人＋AI 的共識與人的定案主權」。

### 根本原則一：一個概念，多個投影
<!-- 推導自：無（這是根公理本身）-->
同一個真理可以、且應該被多個視角看；視角是 perspectives-on-the-whole，不是 parts（源自 John Frame 的三視角，但承重牆是經驗，不是 Frame——見 [history/001-從自動到人在環](history/001-從自動到人在環.md) 旁的 experience 教訓）。

- **意義**：知識不靠「分類到單一位置」組織，靠「概念 + 它在各視角的投影」組織。
- **如何應用**：整理時保持概念完整，把它投影到各視角，而不是把它切碎分進不同資料夾。

### 根本原則二：記憶可（大致）回溯
<!-- 推導自：使命（共識要能被重建）+「code 是 what 的真相」這個事實。
     修訂自舊「knowie 管 why，不管 what」（superseded 2026-06-07，見 history/004）：
     舊表述「完全不管 what」與「可回憶單元需 why+how+概念」衝突，鬆綁為「回溯所需」。 -->
knowie 記 why 為主，並保留**足以讓過去認知被大致重建的最小 how/what**（可回憶單元＝為何＋怎麼＋用了哪些概念）；不複製 code 已是真相的 what（冗餘且會與 code 漂移），但不是零 what——標準是「回溯所需」，不是「純 why」。

- **意義**：要能回到當初「為何這樣決定、怎麼做、用了哪些概念」，就得留一點 how/what；凡 code 自己說得清的 what 不重記。
- **如何應用**：記到「未來能大致重建這次認知」為止；超過的 what（API、架構細節）讓 code 說。

## 延伸原則

<!-- 每條標明推導自哪條根本原則。可演化，但不得牴觸根本原則。 -->

### 1. 整理是投影，不是分割
推導自：根本原則一。把一個概念切碎分進不同類型資料夾，就違反了根公理。按概念組織，不按類型。

### 2. 捕捉，不是發明
推導自：根本原則一。概念是長出來的、名字是認出的、整理是事後的——皆 emergent。不預先設計概念，等它從具體中結晶。

### 3. 自我維護必須收斂
推導自：使命（why 要可信）。整理後再跑一次應近乎 no-op（冪等）；概念向少數根收斂。發散的自我維護 = 自動製造熵。

### 4. 事後可審計 > 事前確認
推導自：使命 + 協議定位。一切在 git 裡，AI 自動跑、人事後審/回滾。「自動」該等於「自動 + 留痕」，不是黑箱。

### 5. 可逆的自動化，不可逆的要人
推導自：使命。搬運/索引/修剪全自動；定案、修訂根本原則、認定母概念是不可外包的人類判斷。

### 6. 記錄轉移，不只記錄狀態
推導自：根本原則二（why 包含「為何變」）。改動留「為什麼變」；舊的標 superseded 並連到新的，不刪。
深層 why：記錄轉移＝把「更新」做成 **append**（不原地改）＝**單調**；這也是 `history` 在多寫者下仍能衍生因果偏序、無需協調的根據（CALM，條件式 payoff）。緣由見 `draft/多agent併發-git模型`。

### 7. 協議，不是平台
推導自：使命（任何 AI 可用）。knowie 不擁有 runtime，只定義「結構 + 行為約定」，寄生在「AI 會讀寫 markdown」這個普世能力上。簡單與工具中立是可攜性的前提，不是美學。

### 8. 連結用標準 markdown；graph 是衍生，不是承載
推導自：根本原則一（一份真相、多個投影 → graph/backlink 是衍生視圖）+ 延伸原則 7（純 markdown、免 resolver、第三方可讀）+ 根本原則二（大規模檢索是 what → 外包後端，不自建）。
- 連結一律 `[]()`（可導航、grep 可驗、免工具）；不用 `[[ ]]`（要 resolver = 假設平台）。
- 雙向/graph **由掃描衍生**（judge 掃正向連結 → 反向索引，與死連結偵測同一動作）或物化少數關鍵節點——不靠 `[[ ]]` 引擎。
- 大規模 graph 檢索不是 knowie 自建；要的話掛載後端（LLM Wiki / 向量庫），adapter 從 `[]()` 結構生 graph。
- 緣由（含 `[[ ]]`/dual/graph-RAG 為何被否決）見 [history/003-連結慣例與檢索定位](history/003-連結慣例與檢索定位.md)。
