# 收斂 vs 發散（Convergence）

> 母概念。自我維護的健康判準。
> ✅ 已釐清（2026-06-07）：**獨立母概念**。理由——[蒸餾](蒸餾.md)（深度）與 [分發非傾倒](分發非傾倒.md)（廣度）都**服務**收斂；被多個機制服務的東西是更上層的目標／判準，不是蒸餾的子性質。蒸餾是「怎麼收斂」的一個手段，收斂是「該長成什麼樣」的判準。

LLM 天生發散（非決定性、漂移）。讓發散的執行者自動維護需要長期一致性的結構，預設結果是熵增。所以 knowie 真正的工程任務，是給發散的執行者套一條**收斂的軌道**。

## 判準
- **冪等性**：整理後再跑一次應近乎 no-op。「再跑會不會動」就是健康檢測。
- **概念數趨平**：概念該向少數根收斂；持續線性爆炸 = 蒸餾欠債的訊號。
- **根本原則低改動率**：根本原則常改 = 混進冒牌貨（分層監控：延伸原則可合理增補）。

## 手段
- 錨點 + 增量（只在邊緣整理，不重排全局）。
- 不變量驅動，不是品味驅動（孤兒偵測、死連結、蒸餾比例）。
- 真投影 / 真獨立 / 假統一：母概念要有剪枝力，否則是壞抽象。
- **獨立收斂 ＝ 真根的證據（三角驗證）**：多條**獨立**推導 / 多面外部鏡子收斂於同一結構 → 它是**真根**而非**假統一**（巧合湊不出多重獨立吻合）。這是上條「怎麼知道是真的」的正向判準。例：[history/004](../history/004-補使命-根公理二鬆綁.md) 多線指向「為共識」、`draft/knowie是一台機器` 三面鏡子（分散式/OS/組合語言）吻合。

## 失敗模式（收斂的反面：怎麼會爛）
收斂失敗 = 知識庫腐爛。watch-list：
- **過度蒸餾** → 失去細節，可回憶單元拼不回現場。
- **根本原則僵化** → 不肯認錯（對策：[讓認錯變便宜](讓認錯變便宜.md)）。
- **人不再審** → 自動化失去人在環、開始漂移。
- **draft 變垃圾場** → 不衰減也不固化，囤積成噪音。
- **judge 當萬靈丹（過度期待）** → 其實 why 對齊靠 next 預防，不是 judge 偵測。
- **過度工程** → 驗證前加沒承諾的功能（已承諾的 roadmap 項不算）。
- **維度坍縮** → dump 進單一檔（對策：[分發非傾倒](分發非傾倒.md)）。
- **母概念肥大（god concept）** → 一個母概念無限吸收，失去剪枝力 → 該分裂出子概念。對策：判準是「它還剪得動嗎」；剪不動就讓子概念自立（現例：[記憶系統](記憶系統.md) 標記觀察中，2026-06-12）。
- **把在場該做的塞給 CLI** → AI 已在場卻發明外部命令 = 多餘 + 逆協議非平台（行為該住可攜 skill，非綁定 runtime）。反覆失敗模式（已四次）；判準見 experience「在場的 AI 自己做」。
- **過時 skill 還可載入執行** → 比過時知識危險（會 acts、可能靜默作惡）；偵測靠「還在用嗎＋用了結果好嗎」（執行＝部分 oracle，見 [why沒有oracle](why沒有oracle.md) 光譜）。

最深的是**認知失調 / 合理化維持**——由 [讓認錯變便宜](讓認錯變便宜.md) 專門對治。judge 是偵測這些的 compliance test；但方向性的爛要靠人。（ROI「值不值」則由真實專案驗，見 vision 拿去真實專案測。）
